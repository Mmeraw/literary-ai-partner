# What Success Looks Like for the Evaluation Process and the Hand-Off to Revision

## Success With IP Protection as a First-Class Constraint

## IP Visibility Classification

This specification describes **internal system architecture** that is protected intellectual property. Sections marked [PUBLIC] describe user-visible surface. Sections marked [PROTECTED] describe internal canon that must never appear in user-facing surfaces, exported artifacts, API responses, error messages, or downloadable reports.

[PUBLIC] Public surface: 13 Story Criteria names, scores, fit/gap framing in plain editorial language, anchored evidence excerpts, plain-language revision priorities.

[PROTECTED] Protected internals: WAVE canon (62 waves, Tsunamis), Gate 15.1/15.2, two-ledger scoring architecture, ritual registry, anchor locks, Lost World doctrine entries, Pass 1/2/3/4 prompt specifications, Volume I–VI references, all wave/gate/doctrine identifiers.

Success now requires a strict separation between what the user sees and what the system uses internally:

#### Tier 1 — Public Surface (User-Visible)

What the user can see, share, screenshot, or export:

- The 13 Story Criteria — names, definitions, scoring bands (1–3, 4–6, 7–8, 9–10). Public IP. Visible.
- Per-criterion scores with fit/gap framing in plain editorial language.
- Anchored evidence excerpts from their own manuscript.
- Reader-impact statements in editorial language ("the reader loses motivation at p.47").
- Revision recommendations in plain craft language ("consider adding a sensory anchor at p.12 to ground the reader").
- Confidence and signal disclosure ("evaluated 78% of your manuscript directly").
- Lifecycle and governance status in user-friendly terms (CERTIFIED, VALID, BLOCKED).
- Prioritized revision queue in editorial language — not wave IDs, not Tsunami names, not canon doctrine references.

#### Tier 2 — Protected Internal IP (User-Invisible)

What the system uses internally but never exposes to the user:

- The 62 WAVE names and structure — Tsunami architecture, wave doctrine, execution order rules.
- Volumes I–VI — all canon documents, doctrine entries, registry IDs.
- Gate 15.1 / 15.2 — implementation details, classification taxonomies, layer architectures.
- Lost World Lessons doctrine — WAVE-31-LW, WAVE-55-L, RITUAL-EDITOR-1, VOICE-LAW-1, ANCHOR-LAW-1, JUDGMENT-LAW-1.
- REC-1A logic, ritual registry, anchor lock, voice mode analyzer, escalation analyzer — all algorithmic IP.
- Pass 1/2/3/4 prompt specifications — verbatim prompts, divergence rules, convergence logic.
- Two-ledger architecture, weighted composite formula, Volume II-A weights — internal scoring methodology.
- Trifecta convergence model, multi-AI orchestration, provider abstraction — Volume V execution architecture.
- SIPOC instrument internals — telemetry field names that reveal architecture.
- Phase 1–6 ritual-aware system, wave safety matrix, wave registry — system internals.
- Binary Acceptance Checklist, V2 Acceptance Doctrine — governance instruments.

## Revised Hand-Off Architecture

### What the user receives at hand-off (translated to public language)

The user gets a revision program expressed entirely in plain editorial language:

> "Based on this evaluation, here are your top 5 revision priorities, in this order. Start with the first one because it stabilizes the foundation your other revisions will build on. For each priority, you'll see what to focus on, where in your manuscript to look, why it matters to your reader, and concrete suggestions for how to revise."

Each priority presented to the user contains:

- A plain-language title — "Strengthen Chapter Endings to Maintain Reader Pull" (NOT "Wave 31 — Scene-to-Scene Pressure Carry")
- A plain-language description — "Your chapter endings need to carry forward unresolved tension rather than settling into resolution. Readers stop reading at chapter breaks where everything feels concluded."
- Anchored evidence from the manuscript
- Concrete revision suggestions in craft language
- Why it's first in dependency-aware editorial language ("Fixing this first will make the next revisions land with more force.")

The user never sees:

- The wave ID
- The Tsunami name
- The canon doctrine reference
- The dependency graph terminology
- The internal classification taxonomy

### What the system uses internally (full IP intact)

Internally, every public statement maps to the protected canon:

| Public surface (user sees) | Internal canon (protected) |
|---|---|
| "Strengthen Chapter Endings to Maintain Reader Pull" | WAVE-31-LW + Wave 31 Scene-to-Scene Pressure Carry |
| "This is your highest priority because it stabilizes the foundation" | Tsunami 1 dependency graph; WAVE-31-LW must clear before WAVE-55-L |
| "We protected your distinctive voice in Chapter 7" | RITUAL-EDITOR-1 ritual registry; REC-1A escalation preservation; anchor locks |
| "Insufficient manuscript evidence to score" | Criterion Observability Model status: NO_SIGNAL / INSUFFICIENT_SIGNAL |
| "Evaluation passed all quality checks" | Gate 15.1 PASS + Gate 15.2 PASS + V2 CERTIFIED + Binary Acceptance Checklist green |
| "Confidence: High" | Confidence Execution Layer derivation; SIPOC instrument; bounded by input scale |

## Revised Success Definition for the Hand-Off

### Machine internals (unchanged in rigor, hardened in opacity)

The full system operates exactly as previously specified — every wave, every gate, every doctrine, every ledger, every governance instrument runs at full strength internally. Nothing is weakened for the sake of the user surface.

### User experience (translated, not exposed)

The user receives the full benefit of the system's intelligence through a public-language translation layer that:

- Renders wave-aligned revision priorities as plain editorial guidance
- Presents the dependency graph as natural revision sequencing rationale
- Shows ritual/anchor protection as "we preserved your distinctive voice in these passages"
- Surfaces governance state as user-friendly status (CERTIFIED, VALID, BLOCKED) without exposing the gate architecture
- Discloses signal sufficiency as plain confidence and coverage language

### The translation layer is itself protected IP

The mapping from internal canon → public language is itself proprietary. The translation rules, the editorial language patterns, the priority justification phrasing — all of it is protected. Competitors seeing the user surface should be unable to reverse-engineer:

- That a 62-wave canonical system exists
- That a Tsunami architecture organizes the waves
- That a paired Gate 15.1/15.2 governs progression
- That a two-ledger scoring architecture separates structural viability from execution readiness
- That a ritual registry and anchor lock system protects voice
- That Lost World doctrine entries govern compression behavior

The user sees a smart, principled evaluation tool. They do not see the methodology that makes it smart and principled.

Export bundles contain only Tier 1 surface content:

- 13 Story Criteria scores and fit/gap analysis (public IP, fully visible)
- Anchored evidence in plain language
- Editorial revision priorities in craft language

Export bundles do not contain:

- Wave IDs, Tsunami names, or canon references
- Gate IDs, doctrine codes, or registry entries
- Internal telemetry field names that reveal architecture
- Prompt specifications or model orchestration details
- Wave dependency graphs or ledger formulas

### What gets persisted internally (full architecture)

The internal artifact bundle preserves everything for audit, replay, regression testing, and future enhancement:

- Full WAVE execution plan with wave IDs and dependency graph
- Complete ritual registry and anchor lock list
- Full SIPOC telemetry with all field names
- Pass 1/2/3/4 raw outputs and convergence reasoning
- Both ledgers with weighted composite formulas

These are accessible only to:

- The system itself (internal use)
- Authorized RevisionGrade engineers (debug, audit, regression)

## Revised Critical Architectural Rule

The single most important rule for IP protection is now:

> No proprietary identifier, doctrine name, registry code, wave number, gate number, Tsunami designation, canon reference, prompt fragment, or internal architectural concept may appear in any user-facing surface, exported artifact, API response, telemetry field exposed to the user, error message, status indicator, or downloadable report.

This must be enforced at:

- The translation layer — a hard boundary function that converts internal artifacts to public surface, with explicit allow-listing of what passes through.
- The export serializer — a sanitization function that strips all internal identifiers before any user-facing payload is produced.
- The error handler — user-facing errors use plain language only ("The evaluation could not complete" not "Gate 15.1 Layer 1 returned FAIL with code Q1 attribution density exceeded").
- The telemetry surface — internal telemetry is logged to internal systems only; any user-visible diagnostic uses plain-language equivalents.
- The CI/governance check — automated linting that scans user-facing strings, response payloads, and exported artifacts for prohibited terms (wave IDs, gate IDs, doctrine codes, etc.).

## The Two Final Rules

1. Public IP gets exposed; protected IP gets used. The 13 Story Criteria are visible. Everything else is internal machinery.
2. The translation layer is the most important protected component. It is what allows the system to deliver the full benefit of the protected canon to the user without revealing the canon itself. It is itself proprietary IP and must be guarded as carefully as the canon it translates.

## Practical Next Step (When You're Ready)

Before any user-facing surface is built or exported, the system needs:

- A sanitization boundary function — applied at every user-facing output point, stripping or translating protected identifiers.
- A CI guard — automated linting that fails any PR introducing protected identifiers into user-facing code paths or strings.
- A public-language style guide — defining the editorial voice and vocabulary used in user-facing surfaces, separate from the internal canon vocabulary.

That translation registry and sanitization boundary become as critical to the system's integrity as the canon itself — because they are what allow the canon to do its work without being seen.

## Extended End-to-End Success Definition (Machine + User)

After everything in memory — the WAVE canon (62 waves, Tsunami architecture), Gate 15.1/15.2, scoring canon (Volume II-A weighted, two-ledger), Criterion Observability/Signal Sufficiency, Runtime Governance v1.0.0, the I47/Chapter 2 contrast benchmark, the V2 Acceptance Doctrine, the Volume III Pass system, the SIPOC instrument, the seed-band governance, the reliability harness, the calibration tooling — the question collapses to this:

> What does it look like, end to end, when the machine has done its job correctly and the user can begin revising with confidence?

### Part 1 — The Machine: What Success Looks Like Internally

#### 1.1 The pipeline runs as a single canonical spine, fail-closed at every joint

Per Runtime Governance v1.0.0, there is exactly one path from manuscript → revision-ready output:

Intake → Queue → Claim → Routing/Chunking → Pass 1 (Structural, deterministic) → Pass 2 (Independent, divergent) → Pass 3 (Convergence, truth resolution) → Pass 4 (Perplexity adjudication) → Gate 15.1 (Dialogue Attribution Purity) → Gate 15.2 (Overcorrection Firewall) → Ledger A (weighted 13-criteria) + Ledger B (WAVE readiness) → Compound Readiness → WAVE Eligibility Decision → WAVE Execution (62 waves, Tsunami sequence) → Revised Manuscript → Binary Acceptance Checklist → Certified Artifact Bundle

Success means every joint between stages is fail-closed. No stage can persist a result if the previous stage's contract was violated. No stage can be skipped, merged, or bypassed.

#### 1.2 The evaluation is constrained by evidence, not schema

Deep success signals from the Criterion Observability Model:

- All 13 canonical criteria are present exactly once.
- Each criterion has a status: SCORABLE | INSUFFICIENT_SIGNAL | NO_SIGNAL | NOT_APPLICABLE.
- NOT_APPLICABLE originates only from governed sources (MDM criteria_plan, evaluation mode), never from the model.
- score_0_10 is number for SCORABLE and null for everything else.
- null and 0 never occupy the same semantic space anywhere in the stack.
- Every scored criterion meets canonical anchor thresholds (point vs pattern criteria distinguished, deterministic distinctness rules enforced).

#### 1.3 The two-ledger architecture is intact

- Ledger A (weighted 13-criteria evaluation) and Ledger B (WAVE execution readiness) are computed separately and never blended.
- Ledger/gate parity invariant holds: buildScoreLedger.wcs === eligibilityGate.computeWeightedCompositeScore for the same input.

#### 1.4 SIPOC instrumentation produces full provenance

Success means an auditor can answer: "what did the evaluator actually read, and how did it represent it?"

#### 1.5 Discrimination, not optimization

- Dense ritual material can correctly yield near-zero compression.
- Transitional material can correctly yield selective compression.

#### 1.6 Gates 15.1 and 15.2 produce clean transitions

Blocking is real, structured reason codes persist, and severity governs continuation.

#### 1.7 WAVE execution respects canonical order + Lost World doctrine

Ritual registry, anchor lock behavior, Tsunami sequencing, and per-wave persistence all hold.

#### 1.8 Confidence is derived, never invented

Confidence is computed from observable penalties and bounded by input scale.

#### 1.9 Certification state is explicit

Release boundary is CERTIFIED | CANDIDATE | REJECTED.

#### 1.10 Revision hand-off is typed and auditable

The boundary payload is deterministic and complete.

### Part 2 — The User: What Success Looks Like Externally

#### 2.1 The user's question is answered fully

What works, what doesn't, where, why, what to fix first, and how to start tonight.

#### 2.2 The user sees a structured evaluation instrument

A clear lifecycle/governance header, 13 criteria with anchored fit/gap, wave-aware revision pathway, top 5 priorities, and evidence/provenance explanations.

#### 2.3 Honest signal sufficiency

NA means insufficient evidence, never a disguised failure score.

#### 2.4 The user receives revision priorities in plain language

Wave internals stay hidden; practical revision guidance stays visible.

#### 2.5 Authorship remains with the user

System proposes; author disposes.

#### 2.6 Timing/progress is honest

No fake progress; async lifecycle state is truthful and resumable.

#### 2.7 The user can act immediately

Output is sufficient for same-night revision kickoff.

### Part 3 — The Hand-Off Joint

#### 3.1 Hand-off is a contract, not a transition

Preconditions (machine + user), immutable payload, and postconditions are explicit and auditable.

#### 3.2 Hand-off is reversible, never silent

Reopen events are logged with causal continuity.

#### 3.3 Philosophical separation is preserved

Evaluation determines manuscript truth; revision determines intervention path.
